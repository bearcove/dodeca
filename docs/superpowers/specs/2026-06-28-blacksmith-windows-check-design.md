# Blacksmith Windows Check Design

## Goal

Add cheap Windows compile signal for dodeca without turning Windows into a release platform yet.

## Current CI shape

`dodeca/.github/workflows/ci.yml` is generated by `cargo xtask ci`; the source of truth is `dodeca/xtask/src/ci.rs`.

The current GitHub workflow does the main validation and artifacts on:

- Linux: `bearcove-ubuntu-24.04`
- macOS: `macos-15`

Windows packaging exists conceptually in installer/release naming, but Windows is not currently part of the emitted target matrix and should not be treated as release-ready.

## Runner policy

- Keep Linux validation on `bearcove-ubuntu-24.04`.
- Add a Windows compile/check lane on Blacksmith using `blacksmith-2vcpu-windows-2025`.
- Do not add Windows packaging, release artifacts, or release dependencies in this change.
- Do not move routine macOS PR checks to Blacksmith; macOS already gets local compile coverage outside CI.
- Keep Blacksmith macOS as a later release-artifact decision, not part of this change.

## Generated workflow change

Add a generated GitHub Actions job named `windows-check` before platform artifact jobs. It should:

1. run on `blacksmith-2vcpu-windows-2025`;
2. check out the repository;
3. install Rust toolchain `1.95`;
4. use the existing Rust cache action path for GitHub workflows;
5. run a Windows-compatible cargo check command.

The first implementation should prefer the broadest useful compile signal. If `cargo check --workspace --all-targets` proves incompatible because the workspace contains Unix-only members, narrow the command at the source to the package set that is expected to build on Windows. The CI job remains a compile check, not a test or packaging lane.

## Verification

- Regenerate GitHub CI from `xtask/src/ci.rs` rather than editing `.github/workflows/ci.yml` by hand.
- Run `cargo xtask ci-github --check` after regeneration.
- Inspect the generated `windows-check` section to confirm the Blacksmith runner label and absence of Windows release dependencies.
